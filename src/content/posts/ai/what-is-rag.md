# RAG 是什么？

对于 RAG 大家应该都不陌生，在目前的技术圈子里面十分火热，今天我们来一起看看 RAG 到底是个什么东西。

RAG 全称 Retrieval Augmented Generation，翻译过来就是检索增强生成，这个名字一眼看上去怪怪的，但我们不用纠结，它实际上就是将一些信息存储起来，等到我们需要一些东西的时候，再给他拿出来。

举个例子，当我们使用一个企业知识库问答平台时，你问了一个问题：我们公司技术部的工作时间是多长时间？

如果没有外部信息，那他回答的结果只能根据训练时数据进行回答。如果接入了 RAG，那么平台构建时会先将公司的各种条例根据 RAG 的流程进行处理并存储起来。等你问出那个问题的时候，先会拿着你的问题根据 RAG 的检索流程去数据库中把需要的东西拿出来，随后进行处理，处理的结果放到大模型的上下文进行拼接。这时候大模型就能回答你的问题了。

RAG 的流程就包括将东西存起来和需要的时候拿出来两个过程。

## 整体流程

RAG 的整体流程如下，这里先简要说明，各位可以先留个印象，详细会在后续说到。

- 数据索引（Data Indexing）：首先对于像是 TXT、PDF、Word 等文档需要先进行处理存储，这叫做数据索引阶段。这个过程中会将文档切分成不同大小的 Chunk，随后将 Chunk 进行向量化，最后将向量化的结果放入到向量数据库中。

- 检索前处理（Pre-Retrieval）：用户提出的问题有时存在口语化、表述不清晰等问题，导致这时如果将其直接和数据库进行相似度计算，会出现查询准确率下降等情况，这时候就需要先将用户问题进行预处理，通常有查询扩充、查询转换等。

- 检索（Retrieval）：这里就开始找出和用户的问题有关联的数据来，需要先将用户问题转为向量，和向量数据库中数据进行相似度比对，获取前几条匹配度最高的。

- 检索后处理（Post-Retrieval）：检索结束后的 Chunks 可能会出现内容相关性差、信息冗余等问题，这时候要进行后处理，会经过过滤、重排序、结果多样化等一系列流程。最后的结果就是检索出的结果了。

## 向量

看了上面提到的整体流程，很多人会存在疑问：为什么要给他转换成向量？还有向量数据库又是啥？相似度是怎么算的？

不急，我们慢慢来，先说为什么要先换成向量，核心原因有三层，本质都围绕语义匹配这件事：

首先，这是大模型的原生语言。所有基于 Transformer 的大模型，输入层都会把文字转成高维向量再做计算，语义理解、逻辑推理全在向量空间里完成。RAG 要在模型外部做文档检索，就必须把文档也转成同规则的向量，保证两边处在同一个语义空间里，匹配结果才能和大模型的理解对齐。

其次，传统关键词匹配只能找字面重合的内容，同义词、不同表述、一词多义的场景都会失效。向量化是按语义相似度匹配，只要意思相关就能命中。

最后，当文字向量化后，一篇文档、一句提问都变成高维空间里的一个点，语义相似度可以直接用空间距离来量化。配合专门的索引算法，就能通过数学运算完成检索。

现在向量我们理解了，那么向量数据库又是什么？

简单说，向量数据库就是专门用来存储、索引高维向量的数据库，它的目标就一个：给定一个查询向量，快速从海量向量里，找出和它空间距离最近、语义最相似的一批结果。

它和传统数据库的本质区别在于查询逻辑完全不同：

- 传统关系型数据库、关键词搜索引擎，做的是精确匹配。

- 向量数据库做的是近似语义匹配，只找语义最像、关联度最高的内容。

而对于相似度的计算，常用的有三种：

- 余弦相似度：含义是看两个向量的夹角大小，夹角余弦值越接近 1 方向就越一致；越接近 0 关联越弱；接近 -1 就越相反。

- 欧氏距离：看两点在空间内的距离远近。

- 点积：既看方向，又将向量本身的大小考虑进去。

## 数据索引阶段

这是 RAG 的第一个阶段，最要做的就是数据构建，要考虑如何使构建的向量更容易被检索。

### Chunk 切分

这里我们首先要说的就是如何将文档进行切块。

这里会分成两个维度，首先是最简单的，按照向量模型的最大词元限制来切分，但是这样切出来的东西会很粗糙。

比如有一句话：我今天吃了包子，喝了豆浆。

如果最大词元是 5，那么就会分成我今天吃了、包子，喝了、豆浆。（仅举例，不严谨）

那切完的话就没有完整语义，所以目前一般不使用这种。

#### Semantic Chunking（语义切分）

按照语义边界来切分文档，不是按照固定字符或 token 数切分，保证每个 chunk 语义完整。

常见做法就是按句子、段落或标题层级切分。

#### Recursive Chunking（递归切分）

按照一组优先级分隔符递归尝试切分。

举个例子：一本小说，我们先按照章节切，再将过长的结果按照段落切分，如果还有长的就按照句号、问号等标点符号切分……

### Chunk 增强

意思就是将切分好的 chunk 增强一下，让他更加便于检索。

#### Contextual Headers（上下文头）

因为 chunk 切分后会导致本来连贯的信息被切断，失去原有语境信息。所以要在 chunk 前补充上下文标题或结构信息，让 chunk 即便脱离原文，也可以保持必要的背景信息。

#### Document Augmentation（文档增强）

在索引前对原始数据文档做增强，就是补充一些额外信息，可以让后续检索效果更好。

例如：文档标题、段落摘要之类的。

#### Metadata Extraction （元数据抽取）

从文档中提取结构化信息，放入 metadata 中，方便后续检索、过滤、排序和展示。

我个人认为元数据十分重要，因为我们可以使用元数据中的字段通过关键词的方式直接对其进行过滤。比如对于公司条例，我要查询关于营销部的，那么如果元数据中有部门字段，我们可以直接将非营销部的进行过滤，效率会很快。

### 索引治理

这里属于索引构建完成后的阶段，相当于数据库里的增删改查阶段。

#### 去重与清洗

对于索引建立文档，我们需要先将其进行标准格式化，将重复、噪音、脏数据清除。

例如：去除重复文档、空白内容、页眉页脚等。

#### Hierarchical Indexing（分层索引）

这是一种索引的建立方式，这里的分层是指索引建立时按照层级关系建立索引，比如说按章节、段落、句子的关系建立。检索时可以先在高层进行检索，随后逐层深入。

#### 增量更新

这属于索引的生命周期管理阶段，是指索引每次不必全量重建，只处理新增、修改或删除的部分，通常依赖其他部分例如：文档版本号、更新时间、内容的哈希值等。

## 检索前处理阶段

这里是为了将用于检索的话进行处理，使其易于检索，总体上称为 Query Transformation。

### Query 改写类

#### Query Rewrite（查询重写）

在不改变用户意图的前提下，把 Query 变得更标准更适合检索。

### Query 扩展类

#### Query Expand（查询扩展）

给原问题添加一些扩展性语句，但通常还是围绕一个检索请求展开。

#### Multi-Query

从不同角度生成多条 Query 分别进行检索，最后再将结果合并。

### Query 结构化类

#### Query Decomposition

把太大、太复杂、不能直接查询的问题拆为多个小的、更易检索的问题。

### HyDE（Hypothetical Document Embedding）

先让模型生成一段假想答案，再使用这个假想答案进行检索。这个不适用于能力较差的模型，由于生成的答案可能是错的，容易导致后续检索出错。

## 检索时阶段

检索时要思考的是如何进行检索才能保证最大程度的准确性。

### Keyword Search（关键词检索）

不用理解整句话的语义，而是看文本里面有没有出现你要找的词。

在真实 RAG 中，可以直接使用成熟的全文搜索引擎，像是 OpenSearch、ElasticSearch 等。

这里扩展一下，这些全文搜索引擎一般基于倒排索引构建，利用 BM25 相关性算法进行检索。

倒排索引是指从词到文档的映射关系；BM25 是用于关键词检索排序的评分函数，用于判断这篇文档和查询到底多相关。

### Vector Search（向量检索）

和关键词检索不同，不看有没有关键词，而是看语义相似度。

而使用的常见方法在上文已经说过，就是向量相似度计算方法的那里，这里不再赘述。

### Hybrid Search（混合检索）

一份 Query 进来后同时做两路检索（关键词和向量），得到的结果后做 RRF 融合。

这种检索方式在实际系统里面更加常用。

#### RRF 融合

根据在不同体系下的排名来计算结果，不关心原始分数是多少，只关心一个文档在各个列表中的排名位置。

举个例子：关键词检索的分数是 0 到 100，向量检索到分数是 -1 到 1。文档 X 在方法 A 中排名第 2，得分 80。文档 Y 在方法 B 中排名第 1，得分 0.95。文档 Z 同时出现在两个列表里，在方法 A 中排名在第 10（得分 30），在方法 B 中排名第 5（得分 0.6）。

在不同体系下的分数是不同的，无法计算，所以要根据排名来计算，这里就会涉及到数学公式的计算了，本篇不会讲的那么细致，不然篇幅过长。

### Metadata Filtering

根据 Metadata 进行过滤，最直接。

## 检索后处理阶段

检索后处理是指拿到了结果后该如何进行结果的处理。

### Result Filtering（结果过滤）

#### Relevance Filtering（相关性过滤）

使用专门的 Reranker 查看 Score，用于比对结果和问题的相关性。

#### Redundancy Filtering（冗余过滤）

查看结果中是否有重复内容。

实现方式可以有：基于 metadata 去重、基于文本相似度去重、基于 Embedding 去重。

#### Conflict Filtering（冲突过滤）

当出现矛盾信息的时候，应该保留哪个。

我们可以通过 metadata 判断哪个更权威或更有时效性，或者交由模型判断。

#### Safety Filtering（安全过滤）

检索到的内容是否适合进入 Prompt。

#### Context-fit Filtering（上下文适配过滤）

有些内容不是完全无关，也无安全问题，但是不值得占用上下文，用处不大。

### Result Ranking（结果排序）

这里可分为两层来看：

1. Re-rank / Reranker：具体的排序动作和实现，可以通过计算 query 和 candidate（候选者）的匹配程度得到新的排序，或者用 Reranker Model 进行打分。

2. Ranking Signals：排序时可参考的信号或特征，可以结合版本、时间等进行参考。

#### Recency Bias（新近性偏置）

当多个候选者内容相关时，是否应该让更新、更可能有效的资料排在前面。

### Result Diversification（结果多样化）

排在前面的结果可能来自同一角度。

这里介绍一下 MMR（Maximal Marginal Relevance，最大边际相关性）：当我们选择下一个 candidate 时，不看他和 query 有多相关，还要看他和已经选择的内容有多重复。

> 公式：MMR = λ \* Sim1(Di, Q) - (1 - λ) \* max(Sim2(Di, Dj))
>
> - **Sim1(Di, Q)** 表示候选内容 Di 与查询 Q 的相关性。
>
> - **Sim2(Di, Dj)** 表示候选内容 Di 与已选内容集合中任意内容 Dj 的相似度。
>
> - **λ** 是调节因子，用于平衡相关性和多样性。
>
> 通过调整 λ 值，可以控制结果更偏向相关性（较高 λ 值）或多样性（较低 λ 值）。

结果多样化就可以通过 MMR 算法来选择最终的 candidate。

## 小结

到这里，RAG 这件事我们就过完了。

如果用一句话概括：**RAG = 检索 + 增强 + 生成**——把外部知识按需检索出来，再塞进大模型的上下文，让模型基于这些信息回答问题。

但也别神化它。RAG 强在能接入实时、专有、不断更新的外部知识；弱在检索质量有上限——如果召回来的不是用户真正想看的，再大的上下文也只是噪声。所以工程上的功夫，大半都花在了检索这一环。

数据治理、检索策略、检索后处理，这三块里任一有短板，整个系统的效果就会大打折扣。
