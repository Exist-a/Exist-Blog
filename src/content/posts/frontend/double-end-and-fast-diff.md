Vue3 把内部的 Diff 算法从 Vue2 的**双端 Diff** 换成了**快速 Diff**（Fast Diff）。这俩到底有啥区别？为什么 Vue3 要换？这一篇我们把它讲明白。

## 一、Diff 是干嘛的

先回答一个最基础的问题：**Diff 算法是什么**。

简单说，Diff 是一种**高效的对比算法**，在 Vue 里专门用于**虚拟 DOM 节点的更新**：当数据变化触发视图重新渲染时，框架需要算出**新旧两棵虚拟 DOM 树之间到底改了哪些地方**，再把那些变化反映到真实 DOM 上，从而尽可能少地操作真实 DOM。

为什么不直接改真实 DOM？因为每动一下真实 DOM 都会触发浏览器的重排重绘，性能开销不容小觑。Diff 算法在 JavaScript 层面就把**哪些要改、哪些不用改**算清楚了，最后只把必要的改动应用到真实 DOM 上。

## 二、虚拟 DOM

那虚拟 DOM 又是什么？

顾名思义，它是**对真实 DOM 的一种抽象**：用 JavaScript 对象来描述一棵 DOM 树的结构。Vue3 里用 **VNode**（虚拟节点）这个概念来表示，一个 VNode 对象中会包含标签名、属性、子节点等真实 DOM 节点对应的关键信息。

举个最简单的例子，下面这段 HTML：

```html
<div class="box">hello</div>
```

在 Vue 内部大概会被表示成这样一个 VNode：

```js
{
  type: 'div',
  props: { class: 'box' },
  children: 'hello'
}
```

所谓 Diff，本质上就是拿着新旧两棵 VNode 树做对比。

## 三、双端 Diff（Vue2）

理解了前置概念，我们来看 Vue2 用的**双端 Diff**。

为什么叫双端？因为这个算法**同时从新旧节点列表的头和尾两边下指针**，通过四个指针的交叉比较，尽可能找到能复用的节点。

### 3.1 四个指针

Vue2 给新旧两组子节点各设了两个指针：

- **旧节点**：`oldStartIdx`（旧头）+ `oldEndIdx`（旧尾）
- **新节点**：`newStartIdx`（新头）+ `newEndIdx`（新尾）

每一轮比对，会依次做下面四种比较：

| 序号 | 比较 | 命中后指针变化 |
|---|---|---|
| 1 | 旧头 vs 新头 | `oldStartIdx++`、`newStartIdx++` |
| 2 | 旧尾 vs 新尾 | `oldEndIdx--`、`newEndIdx--` |
| 3 | 旧头 vs 新尾 | `oldStartIdx++`、`newEndIdx--` |
| 4 | 新头 vs 旧尾 | `newStartIdx++`、`oldEndIdx--` |

只要其中一种比较命中（即两边节点类型和 key 都相同），就立即 patch 一波，然后按上表调整指针，进入下一轮。

### 3.2 四种比较都不命中怎么办

如果四种比较都没命中，框架会退而求其次——用 **key** 在旧节点里直接查这个新头节点存不存在：

- **找到了**：把它 patch 一下，顺手把旧节点列表中对应位置置空（避免下一轮再次被匹配到），然后 `newStartIdx++`
- **没找到**：说明这是个全新的节点，直接挂载到对应位置

### 3.3 何时结束

当 `oldStartIdx > oldEndIdx` 或 `newStartIdx > newEndIdx` 时，主循环结束。这时看哪边还有剩余：

- **新节点还有剩余**：这些是新增的，挂载到 DOM 上
- **旧节点还有剩余**：这些在新列表里已经不存在了，卸载掉

### 3.4 双端 Diff 的优缺点

双端 Diff 巧妙就巧妙在它利用了**真实场景中节点移动的常见规律**：很多时候列表的更新无非就是头部加一个、尾部追加一个、或者整体反序，这几种情况四种比较都能直接命中，效率非常高。

但它也有局限——当节点移动比较复杂、四种比较全 miss 时，就得退化到 key 查找；而且对于**前后都变了、中间部分顺序也变了**这类情况，它并不能保证**移动次数最少**。

## 四、快速 Diff（Vue3）

为了补上双端 Diff 的这些短板，Vue3 借鉴了 [inferno](https://github.com/infernojs/inferno) 这套虚拟 DOM 库的思路，换成了**快速 Diff**。

快速 Diff 的核心思想一句话就能概括：**先把头部和尾部相同的部分预处理掉，再处理中间乱序的部分，并用最长递增子序列来最小化 DOM 移动次数。**

### 4.1 头部预处理

从前往后，比较新旧节点的头：

- 节点类型和 key 相同 → patch，新旧头指针都往后走
- 不相同 → 停下来

这一步结束后，**前面这一段完全相同的节点就被处理掉了**，根本不进入复杂的 Diff。

### 4.2 尾部预处理

紧接着从后往前，比较新旧节点的尾：

- 节点类型和 key 相同 → patch，新旧尾指针都往前走
- 不相同 → 停下来

经过头尾两步预处理，剩下的就是**真正发生变化的中间部分**。

### 4.3 处理中间部分

这是快速 Diff 最精彩的地方。设此时剩余的新旧节点区间分别是 `old[oldStart..oldEnd]` 和 `new[newStart..newEnd]`，处理流程如下：

1. **为新节点建立 key → index 的索引表**：方便后面用 key 直接定位。
2. **遍历旧节点**，对每一个旧节点，去新节点的索引表里查它的 key：
   - **找到了**：说明这个节点在新列表里还在，patch 一下，并把它的旧索引记到一个 `source` 数组里
   - **没找到**：说明这个节点在新列表里已经被移除了，直接卸载

3. **构造 source 数组**：`source[i]` 表示新节点列表里第 `i` 个节点对应的是哪个旧节点。如果某个新节点在旧节点里根本不存在（也就是要新增），`source[i]` 标记为 `-1`。

4. **求最长递增子序列**：在 `source` 数组上求一个**最长递增子序列（LIS）**。LIS 里的节点对应的旧索引是单调递增的，意味着它们在新旧两棵树里的**相对顺序没有变**，所以这些节点**不用移动**。

5. **反向遍历新节点**进行最终操作：
   - `source[i] === -1`：新节点，挂载
   - `i` 在 LIS 中：不动
   - 否则：调用 move 把节点移动到正确位置

### 4.4 为什么要用最长递增子序列

这是快速 Diff 最精髓的地方。我们拿一个直观的例子感受一下：

假设 `source = [2, 0, 1]`，意思是新节点 0 对应旧节点 2、新节点 1 对应旧节点 0、新节点 2 对应旧节点 1。

它的 LIS 是 `[0, 1]`（即新节点 1 和新节点 2 这两个），它们的旧索引 `0 < 1` 是递增的，说明相对顺序没变，不用动。只需要把新节点 0 这一个位置移动一下就够了。

如果不用 LIS 而是简单粗暴地**全部重排**，每个节点都要动一次；有了 LIS，**只需要移动那些不在子序列里的节点**，移动次数最少。

这就是 Vue3 把 Diff 换成快速 Diff 的根本原因——**在大规模列表更新场景下，它能把 DOM 操作次数压到理论最小值**。

## 五、两者对比

| 维度 | 双端 Diff（Vue2） | 快速 Diff（Vue3） |
|---|---|---|
| 核心思路 | 四指针交叉比较 + key 兜底 | 头尾预处理 + key 索引 + 最长递增子序列 |
| 对常见更新（头尾追加） | 四次比较能命中，效率高 | 头尾预处理直接吃掉，效率更高 |
| 对复杂乱序 | 退化为 key 查找，移动次数不一定最少 | 用 LIS 保证移动次数最少 |
| 平均性能 | 良好 | 更优（尤其大列表） |
| 实现复杂度 | 较简单 | 较复杂（多了 LIS 这一步） |

需要补一句：两者在时间复杂度的大 O 上**都是 O(n)**，所以小列表上差距其实并不明显。快速 Diff 真正的舞台是**节点数量较多、且更新模式以中间乱序为主**的场景，这时 LIS 带来的移动次数优化收益才会被放大。

## 六、小结

回顾一下这一篇：

- **Diff 是虚拟 DOM 更新的核心**：用最小代价把数据变化反映到真实 DOM
- **虚拟 DOM** 是对真实 DOM 的 JavaScript 对象抽象，Diff 比对的就是 VNode 树
- **双端 Diff**（Vue2）：维护新旧各两个指针，每轮做头头、尾尾、头尾、尾头四次比较，都不命中再用 key 兜底
- **快速 Diff**（Vue3）：头尾预处理 + 中间 key 索引 + 最长递增子序列，保证中间部分移动次数最少
- **LIS 的意义**：找出新旧两棵树中相对顺序没变的节点子集，这些节点不用移动，其余的才需要动

整体上，Vue3 的快速 Diff 在大列表和复杂更新场景下确实更优，这也是 Vue3 性能比 Vue2 强的一个重要原因。
